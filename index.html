Part 0 · Before you write a line

The workshop & the family tree

You can't script a thing until you know where code lives and what it's allowed to touch. Skip this and every future bug becomes "why won't it run" instead of a real problem you can solve.

0.1

What Roblox Studio actually is

Roblox Studio is two tools wearing one coat. It's a builder (you drag in parts, shape a world) and it's a code editor (you write the logic that makes the world do things). A game is just a world plus the scripts that bring it to life.

🧠 Think of it like

A puppet theatre. The parts are the puppets and the set. The scripts are you, pulling the strings. A puppet alone does nothing — the strings are where "alive" comes from.

You don't need to be a great builder to be a great scripter. Plenty of paid commissions are pure logic dropped into someone else's world. Your job in this course is the strings, not the puppets.

🎮 Try it

Open Studio → New → Baseplate. That flat grey world is your sandbox for the rest of this course. Leave it open.

0.2

The Explorer & the hierarchy

On the right side of Studio is a panel called the Explorer. It's a list of everything that exists in your game, arranged as a family tree — parents and children. This tree is the single most important thing to understand, because your code navigates it constantly.

Everything in the tree is an Instance — a generic word for "a thing in the game." A Part is an Instance. A Script is an Instance. A Player is an Instance. They all sit somewhere in the tree, and they all have a parent.

🧠 Think of it like

Folders on a computer. workspace is a folder. Inside it is a folder for your character. Inside that is your Humanoid. To reach something, you walk the path: folder → folder → thing.

In code, you walk that path with dots. workspace.Baseplate means "the child named Baseplate, inside workspace." Each dot is one step down the tree.

-- "Go into workspace, find the child called Baseplate"
local floor = workspace.Baseplate

-- You can go deeper, one dot per step down the tree
local part = workspace.Model.Door.Handle
🔑 Key idea

The Explorer tree and your code are the same thing seen two ways. When you write workspace.Something, you are pointing at a real row in that panel. If the path in your code doesn't match the tree, your code can't find it — that's the #1 beginner error.

0.3

Services & the game object

At the very top of the tree sits game — the root of everything. Directly under it are Services: big built-in containers, each with a job. You'll use a handful constantly:

  • Workspace — the 3D world. Parts, models, the map.
  • Players — everyone currently in the game.
  • ReplicatedStorage — a shared box both the server and players can see (you'll keep important stuff here).
  • ServerScriptService — where your main server scripts live and run.
  • StarterGui / StarterPlayer — templates copied to each player when they join.

You grab a service with game:GetService("Name"). That's the safe, correct way — always.

local Players = game:GetService("Players")
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local ServerScriptService = game:GetService("ServerScriptService")
✅ Do this

Always grab services with :GetService() at the top of your script and store them in local variables. It's reliable, it's what every pro does, and it makes the rest of the script readable.

🧠 Think of it like

A big building. game is the building. Services are the departments — Workspace is the showroom floor, Players is HR, ServerScriptService is the back office where the real machinery runs.

0.4

The three kinds of script (this matters a lot)

There are three script types. Using the wrong one is the cause of a huge share of "why isn't this working" moments. Learn the difference now and save yourself weeks.

  • Script — runs on the server (Roblox's computer). The boss. It owns the truth: health, money, who has what. Lives in ServerScriptService.
  • LocalScript — runs on one player's own device. Handles what only that player sees: their buttons, their camera, their UI. Lives in player containers like StarterPlayerScripts.
  • ModuleScript — doesn't run by itself at all. It's a reusable toolbox other scripts borrow from. Lives in ReplicatedStorage (shared) or ServerStorage (server-only).
🧠 Think of it like

A restaurant. The Script is the kitchen — one place, trusted, makes the real food. LocalScripts are each customer's table — they can press the call button, but they can't cook. ModuleScripts are the recipe book everyone shares.

🔑 Key idea

Anything that must be fair and cheat-proof (money, damage, saving data) goes in a Script on the server. Anything that's just for one player to see or click goes in a LocalScript. This single rule prevents most exploits. We'll go deep on it in Part 3.

0.5

Your first script + reading the Output

Time to make the computer talk back. print() writes a message to the Output window — your window into what your code is doing. Open it via View → Output. You will live in this window. It's how you see your thoughts made real.

-- Put this in a Script inside ServerScriptService, then press Play
print("The server is alive")
print("2 + 2 is", 2 + 2)

Press the Play button. In the Output you'll see your messages appear. That loop — write code, press play, read output — is 90% of what scripting feels like, forever.

⚠️ Common mistake

If nothing prints, check: is it a Script (not LocalScript) and is it inside ServerScriptService? A LocalScript in ServerScriptService simply won't run — wrong type, wrong place.

🎮 Try it

Make the script print your name and your favourite number. Then make it print 10 * 10. Watch the Output. You just ran code.

Check yourself: where does a Script run, and where does a LocalScript run?

A Script runs on the server (Roblox's computer) — it's trusted and owns the real game state. A LocalScript runs on one player's own device — it only handles what that single player sees or controls. Mixing these up is the root of most beginner bugs.

Part 1 · The Luau language

The grammar of everything

Luau is the language Roblox uses. This part is the alphabet and grammar — variables, decisions, loops, functions, tables. It's maybe 80% of every script ever written. The reason commissions feel impossible right now is that this faded. Rebuild it and the gap mostly closes.

1.1

Variables — labelled boxes

A variable is a named box that holds a value. You put something in, give the box a name, and later you use the name to get the thing back. That's it.

local playerName = "Timz"
local age = 18
local isClimbing = true

print(playerName)   -- Timz
print(age)          -- 18

The word local means "this box exists only in this area" — we'll explain why that matters in 1.8. For now: always write local in front of new variables. It's a habit that prevents real bugs.

🧠 Think of it like

Labelled jars on a shelf. age is the label; 18 is what's in the jar. Say "age" and you get 18 back. Put something new in the jar later and the label still works.

You can change what's in the box at any time — just don't write local again, because the box already exists:

local score = 0
score = score + 10   -- now 10
score = score + 5    -- now 15
print(score)         -- 15
1.2

The basic types: nil, boolean, number, string

Every value has a type — a category that decides what it can do. There are four you'll meet immediately:

  • number — any number. 3, -12, 7.5. You can do maths with these.
  • string — text. Always in quotes: "hello", "Door_3". Quotes are how Luau knows it's text, not code.
  • boolean — only two values: true or false. The yes/no of programming.
  • nil — "nothing here." The empty box. When something doesn't exist, it's nil.
local coins = 100        -- number
local title = "Champion" -- string
local hasKey = false     -- boolean
local reward = nil       -- nothing yet
🔑 Key idea

nil is not zero and not "false" — it means absence. When you ask the tree for a child that isn't there, you get nil. Half of all errors are "I tried to use something that was actually nil." Learning to expect nil is a superpower.

🎮 Try it

Make one variable of each type and print them. Then print type(coins) and type(title) — Luau will literally tell you the type of anything.

1.3

Operators & type coercion

Operators are the little symbols that combine values. The maths ones are obvious: + - * /. Luau adds handy shortcuts so you don't repeat the variable name:

local coins = 50
coins += 10   -- same as coins = coins + 10  → 60
coins -= 25   -- → 35
coins *= 2    -- → 70

For text, .. (two dots) glues strings together. This is called concatenation — you'll use it constantly to build messages:

local name = "Timz"
print("Welcome, " .. name .. "!")   -- Welcome, Timz!

local level = 5
print("You reached level " .. level) -- numbers auto-convert to text here

That last line shows coercion: when you glue a number to text with .., Luau quietly turns the number into text for you. Helpful — but don't rely on it the other way (text doesn't become a number automatically; you'd use tonumber("42") for that).

⚠️ Common mistake

Mixing up .. (glue text) and + (add numbers). "5" + 5 is trouble; "5" .. 5 gives "55". Pick the operator that matches what you actually want.

1.4

Strings — working with text

Text comes up everywhere: names, messages, item IDs. Luau gives you a toolbox called string for slicing and shaping it. A few you'll actually use:

local word = "Roblox"

print(#word)                    -- 6  (the # counts characters)
print(string.upper(word))       -- ROBLOX
print(string.lower(word))       -- roblox
print(string.sub(word, 1, 3))   -- Rob  (characters 1 to 3)

-- string.format builds clean text with slots
local coins = 250
print(string.format("You have %d coins", coins)) -- You have 250 coins

The # in front of a string gives its length. You'll see # again with tables (it counts items there too).

🧠 Think of it like

A label maker. string.format is a template with blank slots (%d for a number, %s for text), and you hand it the values to drop in. Cleaner than gluing fifteen .. together.

1.5

Conditionals — making decisions

This is where code stops being a straight line and starts thinking. An if statement runs code only when something is true.

local coins = 120

if coins >= 100 then
    print("You can afford the sword")
end

The part between if and then is a question that's either true or false. If true, the code inside runs. end marks where the "if" block stops. Add else for the other case, and elseif for more cases:

local health = 30

if health <= 0 then
    print("You died")
elseif health < 50 then
    print("Careful — low health")
else
    print("You're fine")
end

The comparison operators that make the questions: == (equal), ~= (not equal), < > <= >=.

⚠️ Common mistake

= puts a value in a box. == asks "are these equal?" Writing if coins = 100 is an error — you meant if coins == 100. One equals = store. Two equals = compare.

🎮 Try it

Make a variable age. Write an if/elseif/else that prints "child", "teen", or "adult" depending on the number. Change the number and re-run to test every branch.

1.6

Logic: and / or / not

Often a decision depends on more than one thing. These three words combine true/false questions:

  • and — true only if both sides are true.
  • or — true if either side is true.
  • not — flips it: not true is false.
local hasKey = true
local coins = 50

if hasKey and coins >= 30 then
    print("Open the door")  -- needs BOTH
end

if coins >= 100 or hasKey then
    print("Allowed in")     -- needs EITHER
end

if not hasKey then
    print("Find the key first")
end
🧠 Think of it like

A nightclub door. and = "ID and over 18" (both or you're out). or = "guest list or a ticket" (either gets you in). not = "not on the banned list."

1.7

Loops — repeating without copy-paste

Computers are brilliant at doing the same thing many times. Loops are how you say "repeat this." There are three shapes; you'll mostly use two.

Numeric for — repeat a set number of times, counting as you go:

-- from 1 to 5, step by 1
for i = 1, 5 do
    print("Count: " .. i)   -- prints 1,2,3,4,5
end

While — repeat as long as something stays true:

local fuel = 3
while fuel > 0 do
    print("Vroom, fuel left: " .. fuel)
    fuel -= 1
end
⚠️ Common mistake — the infinite loop

If a while condition never becomes false, the loop never stops and your game freezes. Always make sure something inside changes toward ending it (here, fuel -= 1). If you forget, the fuel never drops and it runs forever.

The third shape, generic for with pairs/ipairs, is for walking through tables — we'll meet it in 1.11 once you have tables to walk through.

🎮 Try it

Write a numeric for that prints the 7 times table (7, 14, 21 …). Then write a while loop that counts a variable down from 10 to 0.

1.8

Scope — why we always write local

Scope means "where a variable is allowed to be seen." A local variable lives only inside the block it was made in — the function, the loop, the if. Outside that block, it doesn't exist.

local function test()
    local secret = "hidden"   -- exists only inside test()
    print(secret)             -- works
end

test()
print(secret)  -- ERROR: secret is nil out here — it was local to the function

You might wonder: why bother? Why not make everything visible everywhere (a "global")? Because globals are a trap. They leak across your whole game, two scripts accidentally share the same name, and you get bugs that are agony to find. Locals keep each piece of code in its own lane.

🧠 Think of it like

Rooms in a house. A local variable is a private thought you have in the bedroom — it doesn't follow you to the kitchen. Globals are shouting everything through the whole house; eventually two people shout the same word and chaos.

✅ Do this

Put local in front of every variable, every time, unless you have a specific reason not to (you almost never will). It's the mark of someone who knows what they're doing.

1.9

Functions — machines that do a job

A function is a named chunk of code you can run whenever you want. You write it once, then "call" it by name as many times as you like. This is the heart of not repeating yourself.

-- define it once
local function greet(name)
    print("Hello, " .. name .. "!")
end

-- call it whenever
greet("Timz")    -- Hello, Timz!
greet("Sara")    -- Hello, Sara!

The word in the brackets — name — is a parameter: an empty slot the function fills in with whatever you hand it. When you call greet("Timz"), the value "Timz" (the argument) drops into that slot.

Functions can also hand a value back with return:

local function add(a, b)
    return a + b
end

local total = add(3, 4)   -- total is now 7
print(total)
🧠 Think of it like

A coffee machine. Parameters are the buttons (size, type) — the slots you set. You press go (call it), it does its work inside, and return is the cup it hands back. You don't rebuild the machine each morning; you just press the button.

🔑 Key idea

If you ever find yourself copy-pasting the same few lines with tiny changes, that's a function begging to be born. "Write it once, use it everywhere" is the single biggest upgrade from beginner to competent.

1.10

Multiple returns (tuples)

Unlike most languages, a Luau function can hand back more than one value at once. You catch them with multiple variables on the left.

local function getStats()
    return 100, 50, "Champion"   -- three values
end

local health, coins, title = getStats()
print(health)  -- 100
print(coins)   -- 50
print(title)   -- Champion

You'll see this a lot with built-in functions. The most important one, pcall (Part 7), returns "did it work?" plus "the result" together. Knowing functions can return pairs prepares you for it.

1.11

Tables, part 1 — lists (arrays)

So far each box holds one value. But what if you want a list of things — five items in an inventory, all the players, a row of doors? That's a table. Tables are the most important data structure in Luau; almost everything is built on them.

The simplest table is a numbered list, called an array. Curly braces, values separated by commas:

local inventory = {"sword", "shield", "potion"}

print(inventory[1])   -- sword   (Luau counts from 1, not 0!)
print(inventory[2])   -- shield
print(#inventory)     -- 3       (# counts the items)
⚠️ Important — Luau counts from 1

Most languages start lists at 0. Luau starts at 1. The first item is [1], not [0]. Forget this and you'll get the wrong item or a nil. Tattoo it on your brain.

Now the generic for loop finally makes sense — ipairs walks an array in order:

for index, item in ipairs(inventory) do
    print(index .. ": " .. item)
end
-- 1: sword
-- 2: shield
-- 3: potion
🧠 Think of it like

An egg carton. One container, numbered slots, an egg in each. inventory[2] means "open slot 2." The carton holds many eggs under one label — that's the whole point of a table.

1.12

Tables, part 2 — labelled data (dictionaries)

Numbered slots are great for lists. But sometimes you want to label your data — not "slot 1, slot 2" but "health, coins, name." That's a dictionary: a table where each value has a word (a key) instead of a number.

local player = {
    name = "Timz",
    health = 100,
    coins = 250,
}

print(player.name)     -- Timz
print(player.coins)    -- 250

player.coins = player.coins + 50   -- update by name
print(player.coins)    -- 300

This is how you keep related information together in one tidy package. Instead of three loose variables, one player table holds everything about that player. You'll structure almost all real systems this way.

🧠 Think of it like

A form you fill out. Instead of numbered blanks, it has labelled fields: Name ___, Age ___, Email ___. player.health = "read the Health field." Labels are clearer than counting slots when data has meaning.

To walk a dictionary, use pairs (not ipairs — that's only for numbered lists):

for key, value in pairs(player) do
    print(key .. " = " .. tostring(value))
end
-- name = Timz   /   health = 100   /   coins = 300  (order not guaranteed)
🔑 Key idea

Array = numbered list, order matters, walk with ipairs. Dictionary = labelled fields, look things up by name, walk with pairs. Same braces {}, two uses. Knowing which you need is half of structuring data well.

1.13

Tables, part 3 — nesting & the table toolbox

Here's the real power: tables can hold other tables. This lets you model anything — a list of players, where each player is a dictionary of their stats.

local party = {
    { name = "Timz",  hp = 100 },
    { name = "Sara",  hp = 80  },
}

print(party[1].name)   -- Timz  (item 1, then its name field)
print(party[2].hp)     -- 80

for _, member in ipairs(party) do
    print(member.name .. " has " .. member.hp .. " HP")
end

The _ is just a throwaway name for "I don't care about the index here." Use it when you only want the value.

Luau gives you a table toolbox for changing arrays:

local items = {"sword", "shield"}

table.insert(items, "potion")   -- add to the end → {sword, shield, potion}
table.remove(items, 1)          -- remove item 1   → {shield, potion}

print(#items)                   -- 2
🔑 Key idea

Nested tables are how every real system is built: inventories, save data, leaderboards, shop catalogues — all just tables inside tables. If you're comfortable here, you can model anything a commission throws at you. This is a big checkpoint.

🎮 Try it

Build a shop table holding 3 items, each a dictionary with a name and price. Loop through and print "Sword costs 50". This is a real shop catalogue — you'll reuse this shape in Part 5.

1.14

Bonus: type checking (the modern way)

Luau can optionally check your types as you write, catching mistakes before you even run. You add a little : type annotation. It's optional, but it makes big code far safer and is standard on professional work.

local function addCoins(amount: number): number
    return amount + 10
end

-- Studio will warn you BEFORE running if you do this:
addCoins("hello")   -- type error: expected number, got string

As of early 2026, Roblox ships an improved type checker (the "new type solver") on by default in most modes — better at catching real mistakes and quieter about fake ones. You don't have to master types now, but get used to seeing : number, : string, : Player — it's everywhere in good code.

✅ Do this (later)

For your first month, focus on logic — types can wait. But once a system gets big, start annotating your function parameters. It turns "mystery crash at runtime" into "red squiggle while typing."

Part 2 · Making the world react

Touching the world & hearing it back

You know the grammar. Now you'll reach out and change actual things in the game — and let the game tell your code when something happens. This is the moment scripting stops feeling like notes and starts feeling like a game.

2.1

Instances & properties

Remember every thing in the tree is an Instance. Each Instance has properties — settings you can read and change from code. A Part has a Color, a Size, a Transparency, an Anchored on/off. You change a property with a dot and an equals.

local part = workspace.Baseplate

part.Transparency = 0.5
part.Color = Color3.fromRGB(255, 92, 66)   -- red-orange
part.Material = Enum.Material.Neon

That Enum.Material.Neon is an Enum — a fixed menu of allowed choices. Materials, easing styles, key codes: when there's a limited list of valid options, Roblox uses an Enum so you can't pick something invalid. Type Enum. in Studio and autocomplete shows the menus.

You can also make new Instances from scratch with Instance.new:

local newPart = Instance.new("Part")
newPart.Size = Vector3.new(4, 4, 4)
newPart.Position = Vector3.new(0, 10, 0)
newPart.Parent = workspace   -- nothing appears until you set the Parent!
⚠️ Common mistake

A new Instance is invisible until you set its .Parent. Create it, configure it, then parent it last. Forgetting the parent is "I made a part and nothing happened" — it exists in limbo, attached to nothing.

2.2

Events & :Connect — the world's doorbells

So far your code runs top to bottom and stops. But games are about reacting: when a player touches the lava, when a button is clicked, when someone joins. Roblox lets things announce "I just happened!" — these announcements are events, and you listen to them with :Connect.

local button = workspace.Button

button.Touched:Connect(function(otherPart)
    print("Something touched the button!")
end)

Read it slowly: "On the button's Touched event, Connect this function." You're handing the event a function and saying "run this every time it happens." The function inside is your reaction. It waits, silently, and fires whenever the event does — maybe seconds later, maybe never.

🧠 Think of it like

A doorbell wired to an action. :Connect is the wiring. You connect "ring → I open the door." Then you walk away. The wire waits forever; when someone presses the bell, your action fires — without you standing there checking.

🔑 Key idea

This is event-driven programming, and it's how nearly all of Roblox works. You don't constantly ask "did it happen yet?" — you set up reactions in advance and let the events trigger them. Almost every system you build is "when X happens, do Y," wired with :Connect.

2.3

The events you'll use constantly

The function an event hands you can come with information. Touched hands you the part that touched. PlayerAdded hands you the player who joined. A few you'll meet again and again:

local Players = game:GetService("Players")

-- Fires when anyone joins the game
Players.PlayerAdded:Connect(function(player)
    print(player.Name .. " joined!")
end)

-- Fires when a part is touched — gives you what touched it
workspace.Trap.Touched:Connect(function(hit)
    -- find the Humanoid (the thing that has health) on whatever touched us
    local humanoid = hit.Parent:FindFirstChildWhichIsA("Humanoid")
    if humanoid then
        humanoid.Health -= 20
    end
end)

FindFirstChildWhichIsA("Humanoid") safely looks for a Humanoid and returns nil if there isn't one — so the if humanoid then guard stops errors when a random brick touches the trap instead of a player. Expecting nil, again, saving you.

🎮 Try it

Make a Part called Trap. Use the Touched code above to take 20 health when a player walks on it. Press Play, walk your character into it, watch your health drop. You just built a real game mechanic.

2.4

Debounce — stopping the spam

Run that trap and you'll notice a problem: Touched fires many times a second while you stand on it — you lose all your health instantly. The event is too eager. The fix is a pattern called debounce: a simple on/off flag that says "I'm busy, ignore me for a moment."

local trap = workspace.Trap
local onCooldown = false   -- our flag

trap.Touched:Connect(function(hit)
    local humanoid = hit.Parent:FindFirstChildWhichIsA("Humanoid")
    if humanoid and not onCooldown then
        onCooldown = true          -- shut the door
        humanoid.Health -= 20
        task.wait(1)               -- wait 1 second
        onCooldown = false         -- open the door again
    end
end)

Walk through what happens: first touch, flag is false so we proceed, immediately set it true, deal damage, wait a second, set it false. During that second, every extra Touched hits the not onCooldown check and bounces off. One hit per second instead of fifty.

🧠 Think of it like

A turnstile. After one person goes through it locks for a beat, so the crowd can't all pour through at once. The flag is the lock; task.wait is how long it stays locked.

🔑 Key idea

Debounce shows up everywhere — attacks, buttons, purchases, abilities. Any time an action shouldn't repeat too fast, you reach for a flag. It's one of the first "real developer" patterns, and you'll use it for the rest of your scripting life.

2.5

The task library — waiting, the right way

You just used task.wait(1) to pause for a second. The task library is the modern, correct way to deal with timing. The three you'll use:

  • task.wait(n) — pause this code for n seconds, then continue.
  • task.spawn(fn) — run a function alongside the current one, so neither blocks the other.
  • task.defer(fn) — run a function in a moment, after the current work settles.
print("start")
task.wait(2)        -- pause 2 seconds
print("2 seconds later")

-- run something on the side without stopping the main flow
task.spawn(function()
    task.wait(5)
    print("this fires 5s later, but didn't block anything")
end)
print("this prints immediately")
✅ Do this

Use task.wait and task.spawn — never the old bare wait() and spawn(). The old ones are legacy, less accurate, and slower. If you see wait() in an old tutorial, mentally swap it for task.wait().

Part 3 · The two worlds

Client and server — the thing everyone gets wrong

This is the concept that separates people who can build real multiplayer features from people who can't. Get it now, deeply, and you'll dodge the bugs and exploits that trap everyone else. This is your most important conceptual leap so far.

3.1

Why there are two sides at all

A Roblox game doesn't run in one place. It runs in two: the server (one powerful computer Roblox owns, shared by everyone in the game) and each player's client (their own phone or PC). They're separate computers, talking over the internet.

The server holds the real, shared truth — the actual map, everyone's real health, the real scores. Each client holds their own view of it, and handles things only they care about: their camera, their buttons, their screen.

🧠 Think of it like

A board game night. The board in the middle is the server — one shared truth everyone sees and trusts. Each player's hand of cards is their client — private, theirs to look at. You can rearrange your own hand all you like, but you can't just reach over and change the board; the table has to agree.

When the server changes something — moves a part, updates health — it automatically tells every client, and they update their view. This automatic sync is called replication. Server changes flow down to everyone. That's the default direction, and it matters enormously.

3.2

Server authority (and why clients can't be trusted)

Here's the crucial rule: changes a client makes to the shared world do NOT travel back up to the server or to other players. If a LocalScript on your machine sets your coins to a million, only you see it. The server's real number is unchanged, and the moment it refreshes your view, your fake million vanishes.

This is on purpose. It's called FilteringEnabled (always on in modern Roblox), and it exists because clients can be hacked. A player can run modified code on their own device. If clients could change the real game, every game would be trivially cheatable.

🔑 The golden rule

Never trust the client. Anything that must be real and fair — money, damage, who owns what, saving data — must be decided and applied by the server, in a Script. The client can ask; only the server decides. Burn this in. It's the foundation of every secure system, and Part 6 is entirely about it.

🧠 Think of it like

A bank. You (client) can request a withdrawal. You cannot reach behind the counter and edit the ledger yourself. The teller (server) checks your balance and decides. Games that let the client edit the ledger get robbed instantly.

3.3

RemoteEvents — talking across the boundary

So the client and server are separate, and the client can't touch the real world directly. How do they communicate? Through a special object called a RemoteEvent — a walkie-talkie between the two sides. You keep it in ReplicatedStorage so both sides can see it.

The client fires the remote to ask for something; the server listens and decides whether to do it.

-- CLIENT (LocalScript): player clicked "Buy". Ask the server.
local remotes = game:GetService("ReplicatedStorage"):WaitForChild("Remotes")
local buyEvent = remotes:WaitForChild("BuyItem")

buyEvent:FireServer("sword")   -- "hey server, I want to buy a sword"
-- SERVER (Script): listen, and DECIDE. Never just obey.
buyEvent.OnServerEvent:Connect(function(player, itemName)
    -- 'player' is filled in automatically and CANNOT be faked
    local price = getPrice(itemName)
    if player.leaderstats.Coins.Value >= price then
        player.leaderstats.Coins.Value -= price
        giveItem(player, itemName)
    end
    -- if they can't afford it, we simply do nothing
end)
🔑 Key idea

Notice the server checks before acting — it looks up the price and the real coin count itself. It never trusts that the client "already paid." The first argument player is injected by Roblox and is genuinely trustworthy; everything after it is the client's claim and must be verified.

⚠️ Common mistake

In the LocalScript, use :WaitForChild() to grab the remote — when a player joins, the remote might not have loaded yet. ReplicatedStorage.Remotes.BuyItem can error with "not a valid member"; :WaitForChild("BuyItem") patiently waits for it to exist.

3.4

RemoteFunctions & UnreliableRemoteEvents

A RemoteEvent is fire-and-forget — you shout and move on. Sometimes the client needs an answer back (e.g. "what's in my inventory?"). For that there's a RemoteFunction: the client asks and waits for the server to return a value.

-- SERVER: define what to return
getInventory.OnServerInvoke = function(player)
    return getPlayerInventory(player.UserId)   -- this value goes back to the client
end
-- CLIENT: ask and wait for the answer
local items = getInventory:InvokeServer()   -- pauses here until the server returns
print(items)

There's also the UnreliableRemoteEvent, for high-frequency updates where it's fine if the occasional message gets dropped — like streaming a position 60 times a second. It's lighter on the network.

✅ Do this

Default to RemoteEvent. Use a RemoteFunction only when you truly need a value back, and only client→server — never make the server invoke the client (a hacked client can freeze and stall your whole server). When in doubt: RemoteEvent.

3.5

BindableEvents — talking on the same side

Remotes cross the client↔server gap. But sometimes two scripts on the same side need to talk — two server scripts, say. For that you use a BindableEvent: same idea, no internet crossing, server-to-server or client-to-client only.

-- One server script fires it
myBindable:Fire("level complete")

-- Another server script listens
myBindable.Event:Connect(function(message)
    print("Got: " .. message)
end)
🔑 Key idea

Quick rule: crossing the client/server gap → Remote. Staying on one side → Bindable. Most beginners never need Bindables much; just know they exist so you don't try to force a Remote where it doesn't belong.

Part 4 · Code that doesn't rot

Organization & object-oriented programming

This is the crux pitch. The "advanced systems" you couldn't read in commissions are mostly just well-organized code using the ideas in this part: modules and OOP. It's the hardest hold on the wall — and it's also your payday. Go slow here.

4.1

ModuleScripts — reusable toolboxes

Imagine you've written a great function for giving coins. You need it in five different scripts. Copy-pasting it five times is a nightmare — fix a bug and you have to fix it five times. A ModuleScript solves this: you write the code once, in one place, and every other script borrows it.

A ModuleScript is different from a normal Script: it doesn't run on its own. It builds up a value (usually a table of functions) and returns it at the end. Other scripts grab it with require.

-- ModuleScript named "CoinManager" in ReplicatedStorage
local CoinManager = {}   -- a table to hold our tools

function CoinManager.give(player, amount)
    player.leaderstats.Coins.Value += amount
end

function CoinManager.take(player, amount)
    player.leaderstats.Coins.Value -= amount
end

return CoinManager   -- hand the whole toolbox back
-- Any other Script can now use it:
local CoinManager = require(game.ReplicatedStorage.CoinManager)

CoinManager.give(player, 100)
CoinManager.take(player, 30)
🧠 Think of it like

A shared toolbox in a workshop. Instead of every worker buying their own hammer, there's one toolbox everyone borrows from. Need a better hammer? Upgrade it once, in the box, and everyone instantly has the better one. That's require.

🔑 Key idea

ModuleScripts are how real projects stay sane. Instead of ten giant tangled scripts, you get small focused modules — CoinManager, ShopManager, DataManager — each doing one job. This is the structure every commission expects, and it's surprisingly simple: a table, some functions, return it.

4.2

Abstraction & DRY

Two words pros throw around. They're simpler than they sound.

DRY = "Don't Repeat Yourself." If the same logic appears twice, pull it into a function or module. Repetition isn't just extra typing — it's extra places for bugs to hide and extra work every time something changes.

Abstraction = hiding messy details behind a simple name. When you call CoinManager.give(player, 100), you don't think about how coins are stored — you just say what you want. The "how" is hidden inside, abstracted away.

🧠 Think of it like

Driving a car. You press the pedal (a simple name) and don't think about fuel injection, pistons, timing. The complexity is abstracted behind a pedal. Good code gives the rest of your program simple pedals to press instead of exposing the whole engine.

✅ Do this

When a script gets long and you spot the same pattern three times, stop and ask: "what's the one job here?" Name it, wrap it in a function, and call the name. Your future self reads openDoor() instead of re-deciphering twenty lines.

4.3

Metatables — the secret machinery

This one feels strange at first. Stick with it — it's the engine under OOP, and once it clicks, a lot of "advanced" code suddenly reads as obvious.

Normally, if you ask a table for a key it doesn't have, you get nil. A metatable lets you change that behaviour — it's a hidden helper attached to a table that says "if someone asks for something I don't have, here's what to do instead."

The key piece is __index. It means: "when a key is missing here, go look over there for it." That "look elsewhere for missing things" is the whole trick behind sharing behaviour between objects.

local toolbox = { hammer = "bonk", saw = "zzz" }

local myKit = setmetatable({}, { __index = toolbox })
-- myKit is empty... but its metatable says "missing? check toolbox"

print(myKit.hammer)   -- bonk!  (not in myKit, so it looked in toolbox)
print(myKit.wrench)   -- nil    (not in either)
🧠 Think of it like

Asking your housemate where the scissors are. You check your own drawer (the table) — empty. Your housemate (__index) says "check the kitchen drawer." Missing things get forwarded to a backup place to look. That forwarding is all a metatable does here.

🔑 Don't panic

You won't write raw metatables often. You mostly need to understand __index = "send missing lookups elsewhere," because that's exactly how the OOP pattern in the next section works. Read this twice, then move on — it'll make sense when you see it used.

4.4

OOP — classes, .new, and self

Object-Oriented Programming sounds intimidating; the idea is friendly. You make a blueprint (a "class") that describes a kind of thing — say, a Weapon. Then you stamp out as many real objects from that blueprint as you want, each with its own data, all sharing the same behaviour.

Here's the standard Luau pattern. It uses everything you just learned: a table for the blueprint, __index to share the methods, and a .new function to stamp out objects.

-- ModuleScript "Weapon" — the blueprint
local Weapon = {}
Weapon.__index = Weapon   -- objects will forward missing lookups here

-- the "constructor": builds one new weapon object
function Weapon.new(name, damage)
    local self = setmetatable({}, Weapon)
    self.name = name
    self.damage = damage
    return self
end

-- a "method": behaviour every weapon shares
function Weapon:attack(target)
    target.Humanoid.Health -= self.damage
    print(self.name .. " hit for " .. self.damage)
end

return Weapon
-- Using the blueprint to make real, independent weapons:
local Weapon = require(game.ReplicatedStorage.Weapon)

local sword = Weapon.new("Sword", 25)
local bow   = Weapon.new("Bow", 15)

sword:attack(enemy)   -- Sword hit for 25
bow:attack(enemy)     -- Bow hit for 15

The magic word is self. Inside a method, self means "the specific object this was called on." When you do sword:attack(), self is the sword (damage 25). When you do bow:attack(), self is the bow (damage 15). Same method, different object, different data. The colon : is what quietly passes self in.

🧠 Think of it like

A cookie cutter. Weapon is the cutter (the shape/blueprint). .new() stamps out a cookie. Each cookie is its own object you can decorate differently (different name, damage) — but they all came from one cutter and all behave the same way. self is "this particular cookie."

4.5

Inheritance — building on a blueprint

Sometimes you want a specialised version of a blueprint. A Sword is a Weapon but with extra stuff. Rather than rewrite everything, you make the new class inherit from the old one — it gets all the parent's behaviour for free, then adds its own.

local Weapon = require(game.ReplicatedStorage.Weapon)

local Sword = setmetatable({}, { __index = Weapon })  -- Sword falls back to Weapon
Sword.__index = Sword

function Sword.new(damage)
    local self = Weapon.new("Sword", damage)  -- build the base weapon first
    setmetatable(self, Sword)
    self.canBlock = true                       -- add sword-only data
    return self
end

function Sword:block()
    print("Blocked!")   -- behaviour only swords have
end

return Sword

A Sword object can now call :attack() (inherited from Weapon) and :block() (its own). The __index chain forwards missing lookups up to the parent — same forwarding trick from 4.3, now doing real work.

🔑 Key idea

Inheritance is "is-a." A Sword is a Weapon. An Enemy is a Character. When you catch yourself thinking "this is just like that thing but with extras," that's inheritance asking to be used. Don't overuse it early — but recognize the shape.

🎮 Try it

Build the Weapon module from 4.4. Make a sword and a bow with different damage. Make a dummy with a Humanoid and attack it. Watch the health drop differently per weapon. When this works, the "skill gap" is mostly closed.

4.6

When to use OOP (and when not to)

OOP is powerful but it's not the answer to everything. The honest guidance:

  • Use OOP when you have many things of the same kind, each with its own state — enemies, weapons, pets, towers in a tower-defense game. Blueprint + many objects = perfect fit.
  • Skip OOP for one-off logic. A single door that opens doesn't need a class — a simple function is cleaner. Wrapping everything in objects when you don't need to is its own kind of mess.
✅ Do this

Ask: "will there be lots of these, each tracking its own data?" Yes → class. No → just a function or module. Most commissions that ask for "advanced systems" (mob systems, ability systems, inventory items) are screaming for OOP — which is exactly why learning it unlocks that work.

Check yourself: what does self refer to inside a method?

self is the specific object the method was called on. If you call sword:attack(), self is the sword (with the sword's own data). If you call bow:attack(), self is the bow. One shared method, but self lets it act on each object's individual data. The colon : is what passes self in automatically.

Part 5 · Real game systems

The systems commissions are made of

Everything so far was the toolkit. Now you assemble it into the actual deliverables people pay for: saving data, currency, shops, combat, inventory, UI. These are your portfolio pieces — and the exact things you said clients "always ask for."

5.1

Leaderstats & currency (server-owned)

leaderstats is a special folder Roblox watches: anything you put in it shows up on the player list automatically. It's also where you store currency. The non-negotiable rule from Part 3 applies hard here: the server owns it. Create it in a Script, never a LocalScript.

-- Script in ServerScriptService
local Players = game:GetService("Players")

Players.PlayerAdded:Connect(function(player)
    local leaderstats = Instance.new("Folder")
    leaderstats.Name = "leaderstats"   -- the exact name matters
    leaderstats.Parent = player

    local coins = Instance.new("IntValue")  -- holds a whole number
    coins.Name = "Coins"
    coins.Value = 0
    coins.Parent = leaderstats
end)

Now player.leaderstats.Coins.Value is the player's coin count, owned by the server, shown on the leaderboard, and ready for a shop to spend. An IntValue holds whole numbers; there's also StringValue, BoolValue, etc. for other types.

🔑 Key idea

Because the server creates and changes these, a hacker editing their local copy changes nothing real. This is the golden rule paying off: own the currency on the server, and "give myself a billion coins" exploits simply don't work.

5.2

DataStores — saving & loading (your #1 portfolio skill)

Coins that vanish when a player leaves are useless. DataStores save data permanently, so it's there next time. This is the single most valuable beginner skill to nail cleanly — most newcomers fear it, so doing it right is instantly quotable work.

Two rules before any code. First: DataStore calls talk to the internet, so they can fail — you must wrap every call in pcall (the "try, and tell me if it broke" tool). Second: only store plain data — strings, numbers, booleans, and plain tables. Never store Instances or functions; they silently disappear.

local DataStoreService = game:GetService("DataStoreService")
local playerStore = DataStoreService:GetDataStore("PlayerData_v1")

local DEFAULT_DATA = { coins = 0, level = 1 }

-- LOAD: pcall protects us if the request fails
local function loadData(player)
    local key = "player_" .. player.UserId
    local success, data = pcall(function()
        return playerStore:GetAsync(key)
    end)

    if success then
        return data or DEFAULT_DATA   -- new player? use defaults
    else
        warn("Load failed for " .. player.Name)
        return nil   -- signal failure — do NOT silently hand out defaults
    end
end

GetAsync(key) fetches whatever was saved under that key (here, a key built from the player's unique UserId). New player → it returns nil → we hand them the defaults.

⚠️ Critical mistake — handing defaults on failure

If a load fails (network hiccup), do not give the player default data — you'd be wiping their real progress with zeros, and then saving those zeros over their save. Return nil and either retry or kick them with "please rejoin." A failed load and a new player look similar but must be treated completely differently.

5.3

Saving safely: SetAsync, retries, and BindToClose

Saving mirrors loading: wrap it in pcall. Use SetAsync to overwrite with the latest data. And you must save when the player leaves and when the server shuts down — miss either and you lose data.

local sessionData = {}   -- [userId] = their data while they're playing

local function saveData(player)
    local data = sessionData[player.UserId]
    if not data then return end
    pcall(function()
        playerStore:SetAsync("player_" .. player.UserId, data)
    end)
end

Players.PlayerAdded:Connect(function(player)
    local data = loadData(player)
    if data then
        sessionData[player.UserId] = data
    else
        player:Kick("Couldn't load your data. Please rejoin.")
    end
end)

-- Save when they leave
Players.PlayerRemoving:Connect(function(player)
    saveData(player)
    sessionData[player.UserId] = nil
end)

-- Save EVERYONE if the server is shutting down — easy to forget, vital
game:BindToClose(function()
    for _, player in ipairs(Players:GetPlayers()) do
        saveData(player)
    end
end)
🔑 The session-data pattern

Notice we keep each player's data in a sessionData table while they play, change that as they earn coins, and only write to the DataStore on leave/shutdown. This is the professional pattern: don't hit the DataStore on every coin — Roblox limits how often you can save (and is tightening these limits in 2026). Edit the in-memory copy freely; save it occasionally.

✅ Do this

For real games, add retry logic (try a failed save up to 3 times with a short wait between) and an auto-save loop every couple of minutes as a safety net. Also: Studio has a Data Stores Manager (View → Data Stores Manager) to inspect and edit saves directly while debugging.

5.4

UpdateAsync & Ordered DataStores (leaderboards)

Two more DataStore tools you'll want. UpdateAsync is the safe way to change a value based on its current value — like adding coins — even across multiple servers, with no race conditions. Use it instead of "read then SetAsync" whenever the new value depends on the old.

-- Safely add coins, even if two servers touch the same data
playerStore:UpdateAsync("player_" .. userId, function(current)
    current = current or { coins = 0 }
    current.coins += amount
    return current
end)

For a global leaderboard ("top 10 richest players"), there's a special OrderedDataStore that keeps values sorted for you. Values must be positive whole numbers.

local board = DataStoreService:GetOrderedDataStore("Coins_v1")

board:SetAsync("player_" .. userId, math.floor(coins))   -- record a score

-- read the top 10, highest first
local pages = board:GetSortedAsync(false, 10)
for rank, entry in ipairs(pages:GetCurrentPage()) do
    print(rank .. ": user " .. entry.key .. " — " .. entry.value)
end
🔑 Key idea

Regular DataStore = "remember this player's stuff." Ordered DataStore = "rank everyone." Most games use both: the normal one for each player's save, the ordered one to power a global top-scores board. Now you can build the leaderboards every competitive game shows off.

5.5

Data versioning — future-proofing your saves

A trap that bites everyone eventually: you ship a game, players have saves, then you want to add a new field or rename one. Old saves don't have it. If you're not careful, you crash on every returning player or silently corrupt data.

The fix: tag every save with a _version number, and migrate old shapes forward when you load them.

local CURRENT_VERSION = 2

local function migrate(data)
    local v = data._version or 1
    if v < 2 then
        data.gems = 0          -- new field added in v2
        data._version = 2
    end
    return data
end

-- in loadData, run it: data = migrate(data)
✅ Do this

For a breaking change (totally different shape), also bump the DataStore name itself — PlayerData_v1 → PlayerData_v2 — so old and new never clash. Thinking about versioning early is the difference between a smooth update and a community-wide data-loss disaster.

5.6

Shop systems — putting it together

A shop is your first system that combines everything: a UI button (client), a RemoteEvent (the bridge), server validation (security), and currency (leaderstats). It's the most-requested beginner commission, so build it once and you can sell it again and again.

The flow, end to end: player clicks Buy → LocalScript fires a remote → server checks the price and their real coins → server deducts and grants the item → done. The client only requests; the server decides. (Sound familiar? It's Part 3, applied.)

-- SERVER: the shop's catalogue and the buy logic live here, trusted
local PRICES = { Sword = 50, Shield = 75, Potion = 20 }

buyEvent.OnServerEvent:Connect(function(player, itemName)
    local price = PRICES[itemName]
    if not price then return end            -- unknown item? ignore

    local coins = player.leaderstats.Coins
    if coins.Value < price then return end  -- can't afford? ignore

    coins.Value -= price                    -- charge them
    giveItem(player, itemName)              -- grant the item (server-side)
end)
🔑 Key idea

The prices live on the server, not the client. A hacker can't tell the server "the sword costs 0" — the server looks up its own price table and ignores anything the client claims. Every secure shop works this way. This single pattern (server holds the truth, client only asks) is the backbone of all monetized systems.

5.7

GUI & TweenService — making it feel "paid"

UI is what players actually see and touch. It's built from ScreenGui → Frames → TextButton/TextLabel/ImageLabel. UI lives on the client (it's per-player), so you script it in LocalScripts. A button click is just an event:

-- LocalScript inside the button
local button = script.Parent

button.MouseButton1Click:Connect(function()
    print("Buy clicked!")
    buyEvent:FireServer("Sword")   -- ask the server (back to 5.6)
end)

What makes UI feel professional rather than stiff is animation. TweenService smoothly slides any property from one value to another — a menu gliding in instead of snapping. This polish is often what separates a $10 job from a $50 one.

local TweenService = game:GetService("TweenService")
local frame = script.Parent.ShopFrame

-- "over 0.3 seconds, smoothly, move to the centre"
local info = TweenInfo.new(0.3, Enum.EasingStyle.Quad, Enum.EasingDirection.Out)
local goal = { Position = UDim2.new(0.5, 0, 0.5, 0) }

TweenService:Create(frame, info, goal):Play()

UDim2 positions UI using a mix of scale (a fraction of the screen, so it adapts to any device) and offset (exact pixels). UDim2.new(0.5, 0, 0.5, 0) means "halfway across, halfway down" — centred on any screen. Prefer scale over offset so your UI works on phones and monitors alike.

🎮 Try it

Make a ScreenGui with a Frame off-screen and a button. On button click, tween the Frame into the centre. Then tween it back out. That slide is the difference between "looks made by a beginner" and "looks paid."

5.8

Combat systems — the thing they always ask for

This is the system you said clients always request and you couldn't build. The concept is straightforward once Part 3 is solid: the client requests an attack; the server validates and applies the damage. The client must never deal damage directly — that's how every combat exploit works.

-- SERVER: the only place damage is real
local lastSwing = {}        -- per-player cooldown tracker
local COOLDOWN = 0.6
local DAMAGE = 15
local RANGE = 8

swingEvent.OnServerEvent:Connect(function(player, targetModel)
    -- 1. Cooldown: stop attack spam
    local now = os.clock()
    if now - (lastSwing[player] or 0) < COOLDOWN then return end
    lastSwing[player] = now

    -- 2. Validate the target is real and has a Humanoid
    if typeof(targetModel) ~= "Instance" then return end
    local enemyHum = targetModel:FindFirstChildWhichIsA("Humanoid")
    if not enemyHum then return end

    -- 3. Range check: are they actually close enough to hit it?
    local root = player.Character and player.Character:FindFirstChild("HumanoidRootPart")
    local enemyRoot = targetModel:FindFirstChild("HumanoidRootPart")
    if not root or not enemyRoot then return end
    if (root.Position - enemyRoot.Position).Magnitude > RANGE then return end

    -- 4. Only now, apply damage
    enemyHum.Health -= DAMAGE
end)

Players.PlayerRemoving:Connect(function(p) lastSwing[p] = nil end)

Read the four guards: cooldown (no spamming attacks), target validity (no fake targets), range (can't hit someone across the map), then damage. The client's job was only to say "I swung at this thing." The server checked everything and decided. That's a real, secure combat core.

🔑 Key idea

Every guard answers a way a hacker could cheat: "attack 1000x/sec" → cooldown. "damage a player I can't see" → range. "send garbage to crash you" → type check. Good combat code is mostly defensive checks wrapped around one line of actual damage. This mindset is exactly Part 6.

🎮 Try it

Make a dummy with a Humanoid. Build a tool that fires swingEvent:FireServer(dummy) on activation, and the server code above. Test it — then try to "cheat" by firing the remote rapidly from the command bar. The cooldown should block you. If it does, your system is solid.

5.9

Inventory systems

An inventory is just a table of what a player owns, kept on the server and saved with their data. Everything you learned about tables (Part 1) and DataStores (5.2) combines here.

-- part of the player's saved data
sessionData[userId] = {
    coins = 250,
    inventory = { "Sword", "Potion", "Potion" },  -- an array of item names
}

local function addItem(player, itemName)
    local inv = sessionData[player.UserId].inventory
    table.insert(inv, itemName)          -- add it (5.13)
end

local function hasItem(player, itemName)
    for _, item in ipairs(sessionData[player.UserId].inventory) do
        if item == itemName then return true end
    end
    return false
end
🔑 Key idea

Notice there's nothing new here — it's tables + server ownership + saving, combined. That's the big lesson of Part 5: "advanced systems" aren't new magic, they're the fundamentals assembled. Once the basics are solid, every system is just a recombination. That's why rebuilding the basics was worth it.

Check yourself: why must the shop's prices live on the server, not the client?

Because the client can be hacked. If prices lived on the client, a player could edit their local copy to make everything cost 0 and buy infinitely. With prices on the server, the server looks up its own trusted price table and ignores whatever the client claims — so the exploit does nothing. This is the golden rule (never trust the client) applied to money.

Part 6 · Security

A game vs. an exploited game

You've met the golden rule repeatedly — here it gets its own part, because security is what separates a developer clients trust with their game from one whose game gets ruined in a week. This is a genuine selling point: "I write secure systems" is worth money.

6.1

The one rule, stated plainly

Never trust the client. Every LocalScript runs on the player's own machine, and any player can run modified code. So treat everything a client sends as a potential lie. The server is the referee; the client is a player who might cheat. Decisions belong to the referee.

🔑 The whole of security in one sentence

If something matters — money, items, health, stats, progress — the server must decide it and apply it, after checking it's legitimate. The client may request; it may never command. Internalize this and you've already avoided 90% of exploits.

6.2

Validation: type, bounds, distance

When a remote arrives, the server should interrogate the data before acting. Three checks cover most cases:

  • Type check — is it even the right kind of value? A hacker sends a table where you expect a number to crash your script.
  • Bounds check — is it in a sane range? Clamp it so amount = math.huge can't one-shot everything.
  • Distance check — for world actions, is the player actually close enough? Stops "interact with something across the map."
buyEvent.OnServerEvent:Connect(function(player, quantity)
    -- type
    if type(quantity) ~= "number" then return end
    -- bounds (whole number, 1 to 99)
    quantity = math.clamp(math.floor(quantity), 1, 99)

    local cost = ITEM_PRICE * quantity
    if player.leaderstats.Coins.Value < cost then return end
    player.leaderstats.Coins.Value -= cost
    -- grant items...
end)
✅ Do this

Make validation a reflex: the first lines of every OnServerEvent are guards that bounce bad input. Real attack input is weird on purpose — wrong types, huge numbers, impossible positions. Your guards turn those attacks into harmless no-ops.

6.3

Cooldowns & rate limiting

Even with valid data, a hacker can fire a remote thousands of times a second to spam-buy, spam-attack, or lag your server. The defense is limiting how often each player may act — a cooldown (minimum gap between actions) or a rate limit (max actions per window).

local COOLDOWN = 0.5
local lastUsed = {}

abilityEvent.OnServerEvent:Connect(function(player)
    local now = os.clock()
    if now - (lastUsed[player] or 0) < COOLDOWN then return end  -- too soon? ignore
    lastUsed[player] = now
    useAbility(player)
end)

game.Players.PlayerRemoving:Connect(function(p) lastUsed[p] = nil end)
⚠️ Don't forget cleanup

When a player leaves, clear their entry from tracker tables (lastUsed[p] = nil). Otherwise those tables grow forever with ghosts of players who left — a slow memory leak. Cleaning up after players is a habit worth building early.

6.4

Common exploits & their defenses

The patterns, side by side — what the attacker does, and what stops it:

  • Argument injection (sends wrong types to crash you) → type-check every argument.
  • Damage/value amplification (sends math.huge) → clamp to a sane max.
  • Remote spam (fires thousands/sec) → per-player cooldown or rate limit.
  • Spoofed target (claims to act on another player) → the server resolves identities from its own state, never the client's claim.
  • Duplicate action (replays a valid buy to get two for one) → consume/check state before granting.
🔑 Key idea

Every exploit is the client trying to make the server do something it shouldn't. Every defense is the server checking before it acts. Once you see security this way, you don't memorize a list — you just ask, on every server action, "how could a liar abuse this?" and add the guard. That instinct is what makes you trustworthy with someone's game.

Part 7 · The craft

What separates working code from good code

You can now build real systems. This last part is the craft around them: writing code humans can read, catching errors, hunting bugs, and the patterns pros reach for. It's also where we talk honestly about how to keep this from fading again.

7.1

Naming & readability

Code is read far more than it's written — by clients reviewing your work, by collaborators, and by future you who's forgotten everything. The cheapest way to look professional is clear names.

  • Name things for what they are: playerCoins, not x. isJumping, not flag2.
  • Functions are verbs (they do something): giveCoins(), openDoor(), savePlayerData().
  • Booleans read like questions: hasKey, isReady, canAttack.
  • Use comments to explain why, not what — the code already says what; the reasoning is what gets lost.
✅ Do this

If you have to scroll up to remember what a variable holds, it's badly named. Rename it. Good names are free documentation, and on commission work, readable code is what gets you rehired.

7.2

Error handling with pcall

Some operations can fail no matter how careful you are — DataStore calls, anything touching the network. If you don't handle the failure, it throws an error and stops your script dead. pcall ("protected call") runs risky code and catches the failure instead of crashing.

-- pcall returns: did-it-work? , and the result (or the error message)
local success, result = pcall(function()
    return someRiskyThing()
end)

if success then
    print("Worked! Got:", result)
else
    warn("It failed, but we didn't crash. Error:", result)
    -- handle gracefully: retry, use a fallback, tell the player
end

Remember multiple returns from Part 1.10? This is where it pays off: pcall hands back two things at once — the success boolean and the outcome. You check the first, then use the second.

🧠 Think of it like

A safety net under a tightrope. The risky walk (the DataStore call) might fall. pcall is the net — instead of hitting the ground (crashing your whole script), the fall is caught and you decide what to do next. Always net your network calls.

7.3

Debugging — finding out why it's broken

Your code will break. Constantly. That's not failure — debugging is programming. The skill isn't avoiding bugs; it's calmly finding them. Your tools:

  • print() everywhere — the oldest, best tool. Print values at each step to see where reality stops matching what you expected. "It printed here but not here" pinpoints the break.
  • The Output window — read the red errors. They tell you the line number and the problem. "attempt to index nil" = you used something that was actually nil (almost always a wrong path in the tree).
  • Breakpoints — click the gutter next to a line in Studio and the game pauses there so you can inspect every variable's real value. Powerful once you're comfortable.
🔑 The debugging mindset

When something's wrong, don't guess and flail. Ask: "what do I expect to be true right now?" Then print to check if it actually is. Bugs live in the gap between what you assume and what's real. Print narrows that gap until the bug has nowhere to hide. This patient habit is worth more than any syntax.

7.4

Patterns pros reach for

A few recurring shapes you'll see in good code. You already know the most important one — debounce (2.4). A couple more to recognize:

State machines — when a thing can be in one of several states (a door is Open / Closed / Opening), track the current state in a variable and only allow valid transitions. Stops "open an already-open door" nonsense.

local state = "Closed"

local function toggleDoor()
    if state == "Closed" then
        state = "Opening"
        -- play animation...
        state = "Open"
    elseif state == "Open" then
        state = "Closed"
    end
    -- if it's mid-"Opening", we ignore the press — clean and bug-free
end

Object pooling — creating and destroying objects (bullets, effects) constantly is slow. Instead, make a batch once, hide them when "destroyed," and reuse them. A performance pattern you'll want for anything spawning lots of things fast.

🔑 Key idea

Patterns are just named solutions to problems that come up over and over. You don't memorize them upfront — you hit the problem, learn the pattern, and recognize it next time. Knowing they exist means you'll search "Roblox state machine" instead of reinventing a buggy version.

7.5

Performance basics

You don't need to obsess over performance early — clear, correct code first. But a few habits keep your games smooth:

  • Don't do heavy work every frame unless you must. A loop on RunService.Heartbeat runs ~60×/sec — keep what's inside it tiny.
  • Cache lookups — if you use workspace.Map.Spawn repeatedly, store it in a local once instead of walking the tree every time.
  • Disconnect events you no longer need — :Connect returns a connection you can :Disconnect(). Leaving thousands of dead connections alive leaks memory.
  • Reuse, don't recreate — object pooling (above) over spawning/destroying in hot paths.
✅ Do this

Write it clearly first; optimize only what's actually slow, and use Studio's MicroProfiler (press Ctrl+F6) to find the real bottleneck instead of guessing. "Make it work, make it right, make it fast" — in that order.
